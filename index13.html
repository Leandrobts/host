ANÁLISE: O QUE OS LOGS REVELARAM

🔴 [CRÍTICO] Array.fill() OOB: PATCHEADO neste firmware
oob_idx não encontrado em nenhum dos 48 índices testados — mesmo após pressão GC.
Impacto: o exploit principal do ps4-jsc-fill-oob.html não funciona.
Próximo passo: testar primitivos OOB alternativos (v9 arquivo 01).
🔴 [CRÍTICO] gc() via ArrayBuffer NÃO dispara o GC
Todos os testes de GC mostram alive=100/100 mesmo após 500MB de pressão.
O GC do JSC no PS4 não roda via alocação de ArrayBuffer em loop.
Mas: T04 mostra alive: 50→0 após Promise.then e T13 mostra alive: 20→0 após async/await.
Conclusão: o GC só roda entre microtasks e await suspensions, não por pressão de heap.
Todos os exploits que usam gc() precisam ser reescritos com await.
🟡 [ALTO] Symbol() == Symbol() retorna TRUE
Log 04/T04: ✓ Symbol() == Symbol() true
A spec diz que dois Symbol() diferentes nunca são iguais.
Isso indica um bug na implementação de Abstract Equality para Symbols.
Pode ser explorado para type confusion. Testando mais fundo no v9 arquivo 02.
🟡 [ALTO] ArrayBuffer.transfer disponível (ES2024)
Log 03/T10: ArrayBuffer.prototype.transfer exists: true
Após transfer: buffer original byteLength=0, TypedArray retorna undefined.
new DataView(detached) lança TypeError corretamente.
Mas: o buffer detachado não causa crash — abre superfície para transfer + race conditions.
🟡 [ALTO] Generator yield* usa call stack (não heap)
Log 03/T08: generator 1M yields exc: RangeError (Maximum call stack size exceeded)
Stack máxima: ~10.429 frames. Recursão mútua: ~10.428 frames.
Trampolim funciona para 1M iterações sem stack overflow.
Implicação: yield* recursivo é vulnerável a stack exhaustion igual a recursão normal.
🔵 [MED] DataView: getFloat64(7) retorna valor válido (leitura não-alinhada)
Log 03/T04: getFloat64(7) = 1.7980491576306266e-189
Leituras não-alinhadas são permitidas. Offset 9 já lança RangeError.
getFloat64(NaN) retorna Math.PI (NaN → 0). DataView(buf,16) retorna byteLength=0 (correto).
🔵 [MED] GC funciona via microtask/await — timing determinístico
T04: após 1 Promise.then com gc(64), alive cai de 50 para 0.
T13: após await wait(10) + gc(64), alive cai de 20 para 0.
T05: FinalizationRegistry callbacks chegam todos na mesma latência: 94.8ms (muito determinístico!).
Isso é excelente para exploits: o timing é previsível.
🔵 [MED] eval aninhado falha com SyntaxError
eval("eval(\"eval(42)\")") → SyntaxError (missing ) after argument list)
O escape de aspas dentro de eval não é processado corretamente neste WebKit.
Pode ser explorado para injeção de código via eval em contextos específicos.
⚪ [INFO] Proxy invariants: todos detectados e lançam TypeError
Liar proxy, denier proxy, keyOmitter — todos lançam TypeError corretamente.
O engine verifica invariantes de integridade do Proxy. Não há bypass aqui.
⚪ [INFO] Stack depth: 10.429 frames (LLInt sem JIT)
Sem JIT, o overhead por frame é maior. Com JIT ativo seria ~3-4× mais profundo.
Trampolim com 1M iterações funciona perfeitamente.
⚪ [INFO] ReDoS: (a+)+$ com 20 chars: 88ms
Backtracking catastrófico existe mas é "apenas" 88ms para 20 chars.
Sem timeout automático de regex. Pode causar DoS com strings maiores.
