🔴 Bug #2 — Array.prototype.toReversed() Uninitialized Butterfly Slots

Fonte: qriousec.github.io/post/jsc-uninit · Jan/2024 · Commits 053d9a84 (bug) → 9158c52 (fix)

🔧 ANÁLISE TÉCNICA (qriousec.github.io) — Funciona 100% em LLInt sem JIT BUG: fastToReversed() com array de comprimento 1:
publicLength = 1 (array.length visível)
vectorLength = 3 (optimalContiguousVectorLength → BASE_CONTIGUOUS_VECTOR_LEN = 3)
memcpy copia apenas 1 elemento → slots [1] e [2] ficam UNINIT (0xbadbeef0 ou lixo de heap)

EXPLORAÇÃO EM LLINT:
1. Spray com arrays float [addr_low, addr_high] → popula heap com ponteiros
2. gc() → libera arrays → heap pronto para reuso
3. [-1n].toReversed() → aloca butterfly reusando slots → v2[1] = ponteiro vazado
4. f2i(v2[1]) → endereço de objeto JS → heap leak sem JIT!

FIX (commit 9158c52): zera slots extras após memcpy
STATUS PS4: branch Set/2023 — verificar se commit 9158c52 está incluído
Por que funciona sem JIT: A vulnerabilidade está em fastToReversed() que é C++ puro no runtime do JSC. O bug ocorre antes de qualquer compilação JIT — o array é criado diretamente pela função nativa. Portanto, desativar JIT (como o PS4 faz) não tem impacto na exploitabilidade. A única diferença é que sem JIT, não há aceleração de compilação para a técnica de spray, mas o crash/leak ainda ocorre.
